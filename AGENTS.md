# AGENTS.md

Guidance for AI coding agents working in this repository. Mirrors the architecture notes in [CLAUDE.md](CLAUDE.md) with extra command recipes and style specifics.

## Repository Overview

Docker Compose configurations for 50+ self-hosted services on an x86_64 Linux server. All networking runs through Tailscale. The main orchestration script (`scripts/all-containers.sh`) manages startup, shutdown, health checks, and updates for every container.

Mirrored at [Codeberg](https://codeberg.org/Chris10/do-it-self) and [GitHub](https://github.com/chrisl8/do-it-self/).

## Architecture & Conventions

**One folder = one service = one Docker Compose project.** The folder name becomes the Docker project name. Each folder contains a `compose.yaml` file — it **must** be named `compose.yaml` (not `compose.yml`) or the orchestration scripts will skip it.

### Container Management

- **`restart: on-failure`** — every service uses `restart: on-failure`. This restarts crashed containers automatically but does **not** auto-start on boot (unlike `unless-stopped` or `always`), avoiding collisions with the cron startup script.
- **Enabled state** — managed in `user-config.yaml` (gitignored) and the registry's `default_disabled` flag. Edit via the web admin Configuration tab. There is no enable/disable shell script.
- **`.start-order`** — optional file in a container dir holding an alphanumeric priority (e.g. `000`, `010`, `z010`). Lower values start first; containers without this file start after ordered ones. Stop order is reversed.
- **`mount-permissions.yaml`** — optional per-service file specifying directory ownership/mode applied before container start.

### Ephemeral Container Folders

Container directories at the platform root (`~/containers/<name>/`) contain only compose configuration and platform metadata — **never** user state. All persistent data lives elsewhere: volume data in `~/container-mounts/`, credentials in `~/credentials/`, user overrides in `user-config.yaml`. A container directory can be deleted and recreated from a module without losing any user data.

Files preserved during module update (never overwritten from the module source):

- `config-personal/` — user's config overrides
- `compose.override.yaml` — hardware-specific overrides (e.g. GPU passthrough)
- `.env` — generated by `generate-env.js`

Tailscale node identity lives outside container dirs at `<mount[0]>/tailscale-state/<container-name>/`, referenced via the `TS_STATE_HOST_DIR` env var in each container's `.env`.

### Module System

Containers can come from **modules** (git repos cloned into `.modules/`) or be **personal** (created directly by the user). The `source` field in `container-registry.yaml` tracks provenance: `source: "<module-name>"` for module containers, `source: personal` for user-created ones. See [docs/MODULES.md](docs/MODULES.md) for the full design. CLI: `scripts/module.sh` (subcommands: `add-source`, `remove-source`, `install`, `uninstall`, `update`, `list`, `regenerate-registry`, `dev-sync`).

### Credentials

- Credentials go in `~/credentials/<container-name>.env`, symlinked as `.env` in each service folder: `ln -s ~/credentials/<service>.env .env`.
- The web admin and `scripts/generate-env.js` pull secrets from Infisical at runtime; shell scripts that need secrets follow the `load_secret()` pattern in `scripts/borg-backup.sh:40-64`.
- Environment variables use `VAR=${VAR}` in the compose `environment:` section — no `env_file:` directive. This keeps variables explicit per-container.
- Sensitive values (passwords, API keys, personal URLs) go in `.env`; generic config goes directly in `compose.yaml`.

### Compose File Patterns

- Run as `user: 1000:1000` when the container supports non-root; otherwise leave as default.
- Never combine `image:` and `build:` in the same service — use one or the other.
- No `version:` key — it's legacy in Compose v2.
- No `:latest` tag (it's the default).
- Healthchecks on every service: typically `start_period: 120s`, `start_interval: 5s`, `interval: 5m`.
- Homepage dashboard integration via labels: `homepage.group`, `homepage.name`, `homepage.icon`, `homepage.href`. Group name is injected via `${HOMEPAGE_GROUP}` from the registry.
- Tailscale sidecar pattern: separate `ts` service with `network_mode: service:ts` on the app container.

### External Git Repos

Some services build from cloned git repos (e.g. `tsidp/tsidp/`, `valheim/valheim-server-docker/`, `minecraft/docker-minecraft-bedrock-server/`). Repo URLs, branches, and shallow-clone flags are defined in `container-registry.yaml` under each container's `git_repos` field. Cloned automatically by `setup.sh`; updated via `all-containers.sh --update-git-repos`. The helper `scripts/list-git-repos.js` reads the registry and outputs repo metadata.

## Key Scripts (scripts/)

| Script | Purpose |
|--------|---------|
| `all-containers.sh` | Start/stop/update all containers. Flags: `--start`, `--stop`, `--get-updates`, `--update-git-repos`, `--restart-unhealthy`, `--container <name>`, `--mount <path>` |
| `module.sh` | Install, uninstall, update, and manage module sources. Wraps `.modules/` operations |
| `dev-sync.sh` | Sync live edits back to a module repo (wrapper for `module.sh dev-sync`) |
| `system-cron-startup.sh` | Runs at boot via cron — stops stale containers, starts everything |
| `system-graceful-shutdown.sh` | Ordered shutdown then OS halt/reboot |
| `system-health-check.sh` | Periodic health monitoring, restarts unhealthy containers, pings healthchecks.io |
| `start-web-admin.sh` | Manages the PM2-based web admin dashboard |
| `update-containers-from-diun-list.sh` | Processes DIUN-detected image updates |
| `setup.sh` / `setup-infisical.sh` / `setup-borg-backup.sh` | First-time install on bare Ubuntu and Infisical/Borg bootstrap |
| `hetzner-test.sh` / `test-fresh-install.sh` | End-to-end testing on a fresh Hetzner VPS |

## Build, Lint, and Test

### Python (valheim/valheim-server-docker/env2cfg)

```bash
cd valheim/valheim-server-docker/env2cfg
pip install tox
tox                              # all tox environments
tox -e py3-syntax                # flake8 syntax checking
tox -e py3-tests                 # pytest with coverage
tox -e py3-black                 # black format checking
pytest test/test_env2cfg.py -vv  # single test file
pytest --cov=env2cfg             # coverage report
black --check --diff .           # format check outside tox
```

### Node.js (web-admin)

```bash
cd web-admin
npm run install:all              # install backend + frontend deps
npm run dev:backend              # Express with watch mode
npm run dev:frontend             # Vite dev server
npm run build                    # production frontend build
npm start                        # production backend
```

Web admin runs under PM2 in production (see `scripts/start-web-admin.sh`) — it is not a Docker container.

### Node.js (caddy/Witchazzan)

```bash
cd caddy/Witchazzan
npm install
npm run client                   # Vite dev server
npm run server                   # PM2 dev server
npm run build                    # production build
npx eslint .
npx prettier --check .
```

### Docker Services

```bash
cd <service-directory>
docker compose up -d             # start
docker compose down              # stop
docker compose pull              # update images
```

Prefer `scripts/all-containers.sh` for anything that touches more than one container.

## Code Style

### Python (env2cfg)

- **Line length**: 88 chars (Black default).
- **Formatting**: Black, no custom config.
- **Linting**: flake8 with ignores `E203`, `W503`.
- **Testing**: pytest with coverage (`--cov=env2cfg`).
- **Naming**: PEP 8 (snake_case for functions/variables, PascalCase for classes).
- **ConfigParser**: use `optionxform = str` to preserve case sensitivity in config keys.

### Node.js / JavaScript

- **Formatting**: Prettier (defaults).
- **Linting**: ESLint where configured (currently caddy/Witchazzan).
- **Type system**: plain JS only — no TypeScript.
- **Modules**: ES modules (`"type": "module"` in `package.json`).
- **Package manager**: npm only (no yarn, no pnpm).

### Shell Scripts

- `#!/bin/bash` (needed for arrays and `[[ ]]`). Do not use `#!/bin/sh`.
- `set -e` for error handling.
- Quote every variable: `"$VAR"`, not `$VAR`.
- Shellcheck clean at warning level; use `# shellcheck source=<path>` with `external-sources=true` for sourced files.

### Docker / Compose Files

- No blank lines, copious comments.
- No `version:` key.
- No `:latest` tag (it's the default).
- `restart: on-failure`.
- Healthchecks on every service.

## Notable Sub-Projects

- **web-admin/** — Full-stack React/Express Docker monitoring dashboard with real-time WebSocket updates. Manages container enable state, module installs, backups, and HTTPS checks. Runs via PM2, not Docker.
- **caddy/Witchazzan/** — Phaser 3 multiplayer game with Socket.io, SQLite, JWT auth. Has Playwright E2E tests.
- **caddy/** — Caddy reverse proxy compose file that bundles multiple sub-services (spacymote, meshtastic, ghost, geomyidae, etc.).
